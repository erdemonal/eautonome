Description

This ontology provides an OWL representation of the Projet Maison Eautonome. It models domestic water uses, quality levels, and greywater reuse processes to reduce water consumption in residential buildings.

Overview

The Eautonome architecture consists of physical water infrastructure and monitoring devices forming an IoT observation system. These components generate observations related to domestic water usage and water quality.

The ontology describes the greywater recycling infrastructure, the sensor network, and the resulting observations. It supports the identification of domestic water usage events, the assessment of sensor status, and the optimization of greywater reuse.

Competency Questions

Queries

ID Natural language question Example
Q1 Which sensor made the observation? The physical device that generated the data. For example, the observation <eau:obsFlowHeaterOutput20230715T0730> is made by <eau:flowmeterHeaterOutput>.
Q2 What physical property is measured? The water quality or flow parameter targeted by the sensor. For example, the property is <s4watr:FlowRate> or <s4watr:Conductivity>.
Q3 Where is the measurement point located and which asset does it measure? The room and water asset targeted by the measurement. For example, <eau:mpBathroomHot> is located in <eau:roomBathroom> and measures <eau:pipeBathroomHot>.
Q4 What is the result of the observation? A numeric value combined with a standardized unit. For example, the result is 4.85 expressed in <unit:L-PER-MIN>.
Q5 What is the timestamp of the observation? The exact moment the observation was produced. For example, recorded at "2023-07-15T07:30:00+02:00"^^xsd:dateTime.
Q6 What is the operational status of each device? The current operational state of a hardware component. For example, <eau:flowmeterBathroomCold> is inactive.
Q7 Which temporal extent and which usage activity characterise an observation collection? The temporal extent is described using sosa:phenomenonTime with OWL-Time, either as an interval or as an instant. The associated activity is specified using eau:hasConsumptionEvent linking to an eau:ConsumptionEvent. For example, <eau:obsColMorningShower20230715> has a time:Interval from 07:30 to 07:45 and links to <eau:eventMorningShower>.
Q8 Which procedure is used to acquire the observation? The automated routine executed by the hardware. For example, the system applies <eau:eautonomeObservingProcedure>.
Q9 What are the calibration parameters of the sensor? The mathematical coefficients used to convert raw signals. For example, <eau:flowmeterHeaterOutput> uses a slope of 0.957.

Inference

ID Natural language question Example
R1 Which sensors are attached to the system? Sensors are inferred from their hosting relations within the system. For example, <eau:flowmeterHeaterOutput> is hosted by <eau:adcCard1>, which is hosted by <eau:raspberryPi4>.
R2 Which room is concerned by an observation? The room is inferred from the location of the measurement point. For example, the observation targets <eau:roomBathroom>.
R3 Which measurement points are at operational risk? Measurement points are inferred to be at risk based on the operational status of their associated sensors. For example, <eau:mpBasementCold> is at risk because its sensor is inactive.
R4 Which observations belong to the same usage period or activity type? Observations within the same collection have a common sosa:phenomenonTime. Collections linked to the same eau:ConsumptionEvent via eau:hasConsumptionEvent correspond to the same type of domestic water use, such as a shower.
